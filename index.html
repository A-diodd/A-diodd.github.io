线程没有工人，锁没有钥匙：一个底层洁癖者的硬件级拆解

黑盒里的真相

应用层说“线程是工人”，底层没有工人，只有寄存器的快照。

你的 CPU 核心物理上同一纳秒只能执行 4 条指令（4核）。所谓的“开启了 1000 个线程”，是操作系统内核在玩时间切片欺骗。

上下文切换：时钟中断的欺骗艺术

  1. 操作系统的时钟中断器每隔几毫秒触发一次硬件中断
  2. 内核强行剥夺当前正在运行的代码的控制权
  3. 把当前 CPU 寄存器的值（PC、SP、通用寄存器）拷贝到内存的 PCB 中
  4. 从内存里抓出另一个 PCB 的数据，塞回 CPU 寄存器

这就是线程的本质：被反复保存与恢复的寄存器现场。

锁的底层：LOCK 前缀与内存总线封锁

应用层说“锁是钥匙”，底层没有钥匙。

你调用的 Lock()，在 x86 架构下是一条特殊的汇编指令：**LOCK CMPXCHG**

当 CPU 执行这条指令时：

  • 硬件直接向主板上的内存总线发送电信号
  • 物理封锁这块内存区域
  • 其他 CPU 核心的读写请求被硬件弹回

这让我理解了为什么

  1. 为什么锁很重？ —— LOCK 前缀会阻塞内存总线，影响多核并行
  2. 为什么 Python 多线程对 CPU 密集型任务没用？ —— GIL 本质上是一个巨大的内核级锁
  3. 为什么多线程不一定比单线程快？ —— 上下文切换有成本（保存/恢复寄存器，约 1-3 微秒）
